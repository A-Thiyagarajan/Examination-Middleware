Deployment Layer

Render Deployment

Cloud hosting layer that serves the FastAPI backend, frontend portal pages, and public demo URLs for ExamFlow Nexus.

Role In ExamFlow Nexus

  • Render hosts both the FastAPI backend and the frontend pages served by the application.
  • It turns the project from a local development app into a public interview-ready demo.
  • It provides a stable public URL for staff portal, student portal, and LMS demo routes.
  • It manages build execution, runtime process startup, environment variables, and service logs.

Deployment Workflow

[ Push ] -> [ Build ] -> [ Deploy ] -> [ Serve ]
  • Push: project changes are pushed to the connected GitHub repository.
  • Build: Render installs Python dependencies and prepares the service runtime.
  • Deploy: Render starts the FastAPI application with the configured start command.
  • Serve: staff, student, LMS, and portfolio routes are available through the public URL.

Why Render

  • Easy deployment: Render reduces infrastructure work for a Python web app by handling builds, process startup, HTTPS, and service logs.
  • Free tier: the project can be demonstrated publicly without immediate hosting cost.
  • GitHub integration: deployment can follow repository pushes instead of manual file uploads.
  • Environment config: secrets such as JWT keys, database URLs, and Moodle settings can live outside the codebase.
  • Portfolio readiness: interviewers can open the project through a URL instead of running setup locally.

Why Not Alternatives

AWS

AWS is powerful, but it adds infrastructure complexity: IAM, load balancers, EC2/ECS/App Runner choices, networking, logging, and cost controls. That is unnecessary for this portfolio-scale middleware deployment.

Heroku

Heroku has a simple developer experience, but its free-tier availability has become more limited. Render is a better fit for an accessible public demo with straightforward web-service setup.

Project Implementation

  • Service setup: configure a Render web service connected to the project repository.
  • Build command: install Python dependencies from the requirements file.
  • Start command: run the FastAPI app with the host bound to 0.0.0.0 and the port supplied by Render.
  • Environment config: store JWT secret, database URL, Moodle token, and deployment flags as Render environment variables.
  • Static/frontend delivery: frontend templates and static assets are served by the deployed FastAPI application.

Deployment Config Showcase

Render takes the project through a simple lifecycle: build dependencies, inject environment variables, start the FastAPI service, and keep it observable through a health endpoint.

Build -> Deploy -> Run

Render Service Config

render.yaml 
services:
  - type: web
    name: examflow-nexus
    env: python
    rootDir: exam_middleware
    buildCommand: pip install -r requirements.txt
    startCommand: uvicorn app.main:app --host 0.0.0.0 --port $PORT
    healthCheckPath: /health
    autoDeploy: true

What it does: defines the Render web service, points it at the Python app directory, installs dependencies, starts Uvicorn, and configures a health check path.

Why it is important: this is the deployment contract. Render uses it to turn a Git push into a running public service without manual server setup.

Environment Variables Setup

render-env 
DATABASE_URL=postgresql+asyncpg://user:password@host:5432/exam_middleware
SECRET_KEY=render-managed-secret
MOODLE_BASE_URL=https://your-moodle-site.com
MOODLE_ADMIN_TOKEN=server-side-token
DEBUG=false
RELOAD=false
PORT=10000

What it does: injects the database connection, signing secret, Moodle settings, and runtime flags through Render instead of hard-coding them in the repository.

Why it is important: environment variables are what make the same codebase portable across local development and hosted deployment while keeping secrets out of Git.

Start Command

start-command 
uvicorn app.main:app --host 0.0.0.0 --port $PORT

What it does: launches the FastAPI application object from `app.main`, binds it to all interfaces, and listens on the port Render provides at runtime.

Why it is important: this is the handoff from deployment to execution. Once the build completes, Render uses this command to keep the service alive and routable.

Health Check Endpoint

health.py 
@router.get("/", response_model=HealthCheckResponse)
async def health_check(db: AsyncSession = Depends(get_db)):
    await db.execute(text("SELECT 1"))
    is_connected, message = await moodle_client.check_connection()
    return HealthCheckResponse(
        status="healthy" if is_connected else "degraded",
        version=settings.app_version,
        database="healthy",
        moodle_connection=message,
    )

What it does: verifies database connectivity, checks Moodle availability, and returns a structured health response that Render can probe at `/health`.

Why it is important: Render uses this endpoint to decide whether the deployed app is actually alive after startup. It also gives operators a quick signal when deployment succeeded but dependencies are degraded.

Request Flow

[ Browser ] -> [ Render Public URL ] -> [ FastAPI Runtime ] -> [ PostgreSQL / File Storage / Moodle ] -> [ Response ]
  • Render is the hosting boundary; business logic remains inside FastAPI services.
  • Logs from this boundary help diagnose deployment, startup, upload, and integration errors.

Performance

  • Cold start: free-tier services may take time to wake up after inactivity, so the first request can be slower than later requests.
  • Uptime: paid instances or always-on configurations are more appropriate for real exam windows where downtime is unacceptable.
  • Resource limits: CPU and memory limits affect file processing and concurrent upload workloads.
  • External latency: database and Moodle calls still dominate response time if those services are slow.
  • Scaling path: upgrade instance size, add managed PostgreSQL, and move files to durable object storage as usage grows.

Security Considerations

  • Store JWT secrets, database URLs, and Moodle tokens in Render environment variables.
  • Use HTTPS public URLs for login, upload, dashboard, and LMS submission flows.
  • Review logs to ensure credentials, tokens, and full request payloads are not printed.
  • Use least-privilege database credentials for the deployed service.

Limitations

  • Sleeping service: free services can sleep when inactive, causing slower first response times.
  • Limited resources: free or small instances may struggle with many simultaneous uploads or heavy file processing.
  • Ephemeral storage: local disk should not be treated as permanent storage for scanned answer sheets in production.
  • Platform dependency: deployment behavior depends on Render build/runtime rules, logs, and service availability.

Final Justification

Render is the right deployment choice for ExamFlow Nexus because it gives the project a public, low-friction hosting path for both backend and frontend experiences. It is simple enough for a portfolio deployment while still supporting environment configuration, service logs, and an upgrade path for more reliable production hosting.
