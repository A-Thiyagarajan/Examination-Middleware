Authentication Layer

JWT Authentication

Stateless security layer protecting staff upload, student dashboard, and submission APIs in ExamFlow Nexus.

Role In ExamFlow Nexus

  • JWT authentication acts as the API access gate between browser portals and protected FastAPI routes.
  • It identifies whether the caller is staff, student, or another allowed actor before upload, dashboard, or submission logic runs.
  • It protects sensitive exam records such as scanned answer-sheet mappings, register numbers, and LMS submission state.
  • It keeps the backend stateless so multiple API instances can validate requests without shared session storage.

Authentication Workflow

[ Login ] -> [ Generate JWT ] -> [ Attach ] -> [ Verify ] -> [ Authorize ]
  • Login: staff or student submits credentials through the portal.
  • Generate JWT: backend signs a token containing subject, role, and expiration claims.
  • Attach: client sends the token with protected API requests.
  • Verify: FastAPI dependency decodes token, checks signature, algorithm, and expiration.
  • Authorize: role checks decide whether the request can access staff upload, student dashboard, or submit actions.

Why JWT

  • Stateless: the backend does not need server-side session storage for every logged-in user.
  • Scalable: any FastAPI instance with the signing secret can validate the token, which supports horizontal scaling.
  • Secure claims: signed payloads prevent clients from changing role or subject values without invalidating the signature.
  • API-friendly: staff, student, and submission routes can all use the same Authorization-header pattern.
  • Short-lived access: expiration reduces the impact of a leaked token.

Why Not Alternatives

Session Auth

Session authentication is reliable for classic server-rendered apps, but it is stateful. ExamFlow Nexus would need shared session storage or sticky routing when multiple API instances serve upload and dashboard requests.

Cookies Only

Cookies alone are less explicit for API-style authorization. They also require careful CSRF handling and can blur the separation between staff/student APIs and browser session behavior.

Project Implementation

  • Token creation: login endpoints issue signed JWTs after credential verification.
  • Dependency injection: FastAPI dependencies decode tokens and expose validated identity to route handlers.
  • Middleware: request middleware can apply headers, request logging, CORS rules, and centralized error boundaries before protected routes execute.
  • Role checks: staff token is required for /upload, while student token is required for /student/dashboard and /submit.
  • Ownership checks: student token subject is matched against database records so a student cannot fetch another student's artifact.

JWT Security Showcase

The authentication path is intentionally small and stateless: login issues a signed token, the client sends it on each request, and FastAPI dependencies validate it before protected business logic runs.

Login -> Token -> Request -> Validation

Token Creation

security.py 
def create_access_token(data: dict, expires_delta: timedelta | None = None) -> str:
    to_encode = data.copy()
    expire = datetime.now(timezone.utc) + (expires_delta or timedelta(minutes=settings.access_token_expire_minutes))
    to_encode.update({
        "exp": expire,
        "iat": datetime.now(timezone.utc),
        "jti": secrets.token_hex(16),
    })
    return jwt.encode(to_encode, settings.secret_key, algorithm=settings.algorithm)

What it does: signs a JWT with subject data plus expiration, issued-at time, and a unique token identifier before returning it to the authenticated client.

Why it is important: this is the source of stateless authentication. The server does not need to keep a session row for every staff request because the token carries the claims needed for later validation.

Token Validation Dependency

auth.py 
async def get_current_staff(
    token: str = Depends(oauth2_scheme),
    db: AsyncSession = Depends(get_db),
) -> StaffUser:
    payload = decode_access_token(token)
    if payload is None or payload.get("type") != "staff":
        raise HTTPException(status_code=status.HTTP_401_UNAUTHORIZED, detail="Could not validate credentials")
    result = await db.execute(select(StaffUser).where(StaffUser.id == int(payload["sub"])))
    staff = result.scalar_one_or_none()
    if staff is None or not staff.is_active:
        raise HTTPException(status_code=status.HTTP_403_FORBIDDEN, detail="Account is disabled")
    return staff

What it does: reads the bearer token from the request, decodes it, checks the actor type, then resolves the concrete active staff user from the database before the route executes.

Why it is important: FastAPI dependency injection turns token validation into a reusable boundary. Every protected route gets the same auth guarantees without repeating decode logic in controller code.

Protected Route Example

upload.py 
@router.post("/single", response_model=FileUploadResponse)
async def upload_single_file(
    file: UploadFile = File(...),
    exam_session: str = Form(...),
    db: AsyncSession = Depends(get_db),
    current_staff: StaffUser = Depends(get_current_staff),
):
    content = await file.read()
    is_valid, message, metadata = file_processor.validate_file(content, file.filename)
    if not is_valid:
        return FileUploadResponse(success=False, message=message, errors=[message])

What it does: declares a protected upload endpoint where the request is only allowed to reach file-processing logic if `get_current_staff` successfully validates the JWT first.

Why it is important: this demonstrates the full flow in practice: login produces the token, the client attaches it to the request, and FastAPI blocks unauthorized callers before exam artifacts are touched.

Security Analysis

  • Expiration: tokens should expire quickly, especially for public demo portals or shared machines.
  • Encryption vs signing: standard JWTs are signed, not encrypted. Do not place passwords, register numbers, or sensitive artifact paths inside the payload.
  • Validation: always validate signature, algorithm, expiration, subject, and role before route logic runs.
  • Transport security: send tokens only over HTTPS so bearer tokens are not exposed in transit.
  • Secret storage: keep SECRET_KEY in deployment environment variables, not source code.
  • Authorization boundary: JWT proves identity and role, but database checks still enforce artifact ownership.

Performance & Scalability

  • JWT validation is local and fast because it does not require a database or session-store lookup for every request.
  • Stateless verification supports multiple FastAPI workers or future horizontal scaling.
  • Role claims reduce repeated user-role queries, while ownership checks still query PostgreSQL when sensitive records are requested.
  • Short tokens keep request headers small and reduce parsing overhead.

Limitations

  • Token revocation complexity: once issued, a token remains valid until expiration unless a denylist or token version strategy is added.
  • Client storage risk: poorly stored tokens can be stolen by XSS or browser compromise.
  • Payload visibility: JWT payloads are base64url encoded and readable, so they must not contain secrets.
  • Key rotation: changing signing keys requires a planned rollover strategy so valid users are not abruptly logged out.

Final Justification

JWT is the right authentication layer for ExamFlow Nexus because the system needs secure, stateless API access across staff upload, student dashboard, and LMS submission flows. It scales cleanly, integrates naturally with FastAPI dependencies, and keeps authorization close to the routes that protect exam data.
