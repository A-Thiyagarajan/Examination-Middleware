Relational Storage Layer

PostgreSQL

Consistent relational database for users, submissions, artifacts, processing logs, and Moodle synchronization state.

Role In ExamFlow Nexus

  • PostgreSQL is the system of record behind the FastAPI middleware.
  • It stores users, students, staff identities, scanned-paper submissions, artifact references, subject mappings, processing queue records, processing logs, Moodle sync results, and audit logs.
  • It connects the scanned file world to the LMS world by keeping a reliable mapping between uploaded artifacts and verified students.
  • It protects the workflow from partial state because upload, verification, retry, and Moodle submission all require consistent database updates.

Database Workflow

[ Insert ] -> [ Map ] -> [ Join ] -> [ Update ] -> [ Log ]
  • Insert: staff upload creates artifact and submission rows.
  • Map: filename metadata maps the artifact to a student and subject.
  • Join: student dashboard joins users, submissions, artifacts, and subjects to show only authorized records.
  • Update: status changes from uploaded to processed, verified, queued, submitted, or failed.
  • Log: processing logs and Moodle sync logs record every important transition for auditability.

Why PostgreSQL

  • Relational structure: the system depends on relationships between users, submissions, artifacts, subjects, and LMS sync records.
  • Strong consistency: exam records must not point to the wrong student or disappear midway through a Moodle submission.
  • ACID compliance: upload and mapping operations can be wrapped in transactions so partial writes do not corrupt the workflow.
  • Indexing and query planning: dashboard and retry queries can be optimized around student ID, subject ID, status, and timestamps.
  • Auditability: append-style log tables make the project easier to debug and explain in interviews.

Why Not Alternatives

MongoDB

MongoDB document storage is flexible, but ExamFlow Nexus needs relational joins across users, submissions, artifacts, subjects, queue state, and audit logs. Embedding everything would duplicate data and make gradebook reconciliation harder.

MySQL

MySQL could support basic relational storage, but PostgreSQL offers stronger flexibility for future metadata columns, JSON fields, advanced indexing, reporting queries, and constraint-heavy modeling.

Project Implementation

  • Foreign keys: submissions reference users/students, artifacts, and subjects so invalid mappings are rejected by the database.
  • Indexes: lookup-heavy fields such as username, register number, submission status, student ID, and subject ID should be indexed.
  • Queries: dashboard queries join the tables needed to show only a student's own mapped answer sheets.
  • Status updates: queue and Moodle sync flows update deterministic state fields instead of relying on file presence.
  • Logs: processing logs and audit logs make retry failures explainable.

Technical SQL Showcase

ExamFlow Nexus uses relational tables to connect authenticated users, scanned exam artifacts, and submission-state tracking. The workflow depends on those links being explicit and queryable.

Table Schema: Users, Artifacts, Submissions

schema.sql 
CREATE TABLE staff_users (
  id SERIAL PRIMARY KEY,
  username VARCHAR(100) UNIQUE NOT NULL,
  email VARCHAR(255) UNIQUE NOT NULL,
  role VARCHAR(50) DEFAULT 'staff',
  is_active BOOLEAN DEFAULT TRUE
);

CREATE TABLE examination_artifacts (
  id SERIAL PRIMARY KEY,
  artifact_uuid UUID UNIQUE NOT NULL,
  parsed_reg_no VARCHAR(20),
  parsed_subject_code VARCHAR(20),
  file_blob_path VARCHAR(512) NOT NULL,
  workflow_status VARCHAR(32) NOT NULL
);

CREATE TABLE submission_queue (
  id SERIAL PRIMARY KEY,
  artifact_id INTEGER NOT NULL,
  status VARCHAR(20) DEFAULT 'QUEUED',
  retry_count INTEGER DEFAULT 0
);

What it does: defines the core relational building blocks for the workflow: staff identity, scanned exam artifacts, and queued submission state for Moodle delivery.

Why it is important: this structure mirrors the actual project flow where uploads become durable artifacts first, then move through a tracked submission pipeline instead of relying on temporary in-memory state.

Foreign Key Relationships

relationships.sql 
ALTER TABLE examination_artifacts
  ADD CONSTRAINT fk_artifact_uploaded_by
  FOREIGN KEY (uploaded_by_staff_id) REFERENCES staff_users(id);

ALTER TABLE submission_queue
  ADD CONSTRAINT fk_queue_artifact
  FOREIGN KEY (artifact_id) REFERENCES examination_artifacts(id);

ALTER TABLE audit_logs
  ADD CONSTRAINT fk_audit_artifact
  FOREIGN KEY (artifact_id) REFERENCES examination_artifacts(id);

What it does: links uploaded artifacts back to the staff member who created them, then links both retry state and audit history back to the same artifact row.

Why it is important: these relationships preserve traceability. Every queued submission and every audit event can be tied to a concrete scanned paper, which is critical for review, retry, and debugging.

Query Example: Users + Submissions + Artifacts

dashboard-query.sql 
SELECT
  su.username,
  ea.artifact_uuid,
  ea.parsed_reg_no,
  ea.parsed_subject_code,
  ea.workflow_status,
  sq.status AS submission_status,
  sq.retry_count
FROM staff_users su
JOIN examination_artifacts ea ON ea.uploaded_by_staff_id = su.id
LEFT JOIN submission_queue sq ON sq.artifact_id = ea.id
WHERE ea.parsed_reg_no = :register_number
ORDER BY ea.uploaded_at DESC;

What it does: joins the uploader, the artifact record, and the submission queue so the system can show who uploaded a paper, what metadata was parsed, and whether Moodle delivery is pending, retrying, or complete.

Why it is important: this is the exact kind of relational query that supports staff review and student-facing workflow visibility. The database does not just store rows; it assembles the end-to-end paper lifecycle.

Index Example

indexes.sql 
CREATE INDEX ix_artifacts_reg_subject
  ON examination_artifacts (parsed_reg_no, parsed_subject_code);

CREATE INDEX ix_artifacts_status
  ON examination_artifacts (workflow_status);

CREATE INDEX ix_queue_status_retry
  ON submission_queue (status, next_retry_at);

What it does: accelerates the most common access paths: finding a paper by register and subject, filtering artifacts by workflow status, and scanning the retry queue in processing order.

Why it is important: these indexes make the exam workflow practical under load. Upload review, student lookups, and queued Moodle retries stay fast because the database is optimized for the real query patterns.

Performance

  • Indexing: composite indexes on student_id + status and subject_id + status speed dashboard and staff review queries.
  • Query optimization: dashboard queries should select only required columns instead of loading large artifact metadata blobs.
  • Connection pooling: FastAPI should use controlled DB sessions to avoid exhausting database connections during concurrent uploads.
  • Queue queries: retry workers can efficiently scan rows with statuses like queued, failed, or retrying.
  • File separation: PostgreSQL should store file paths or object keys, while binary scanned papers remain in file/object storage.

Security

  • Role access: staff queries can see upload and processing state, while student queries must be filtered by authenticated student identity.
  • Data integrity: foreign keys prevent submissions from referencing missing users, subjects, or artifacts.
  • Constraint safety: non-null status fields and controlled status values reduce invalid workflow states.
  • Secret handling: database credentials should live in environment variables, not frontend code or committed files.
  • Audit trail: processing logs and Moodle sync logs help detect unexpected submission changes.

Limitations

  • Scaling complexity: high-volume exam windows may require connection pooling, query tuning, larger database resources, or read replicas.
  • Migration discipline: relational schema changes need migrations and careful deployment planning.
  • Large file handling: PostgreSQL is not the right place to store large scanned paper binaries.
  • Index tradeoffs: indexes improve reads but can slow high-volume writes if added without clear query patterns.

Final Justification

PostgreSQL is the right database for ExamFlow Nexus because the project is built around reliable relationships: students, uploads, artifacts, subjects, queue state, Moodle syncs, and audit logs. Strong consistency and relational joins make the scanned-paper-to-LMS pipeline trustworthy and explainable.
