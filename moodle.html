Moodle LMS Integration

External academic system where verified scanned-paper submissions are handed off for assignment tracking and gradebook workflows.

Role In Examination Middleware

  • Moodle acts as the external system for final submission and grading.
  • Examination Middleware does not replace the LMS; it bridges scanned exam artifacts into LMS-compatible submission workflows.
  • FastAPI prepares validated records, maps them to assignment context, and sends them to the Moodle integration layer.
  • Moodle becomes the academic destination where faculty can review or grade records using familiar LMS concepts.

Integration Workflow

[ Fetch Assignment ] -> [ Map ] -> [ Upload ] -> [ Submit ] -> [ Grade ]
  • Fetch Assignment: middleware identifies the course/subject assignment target for the scanned paper.
  • Map: student register number, subject code, artifact path, and Moodle assignment ID are linked.
  • Upload: file reference or artifact payload is prepared for Moodle-compatible submission.
  • Submit: Moodle API client sends the request with LMS token authentication.
  • Grade: Moodle assignment/gradebook state becomes the downstream academic record.

Why Moodle

  • Standard LMS: Moodle is widely used in academic environments and matches the project context of exams, students, faculty, assignments, and grades.
  • API support: Moodle can expose web-service style integration points for assignment and gradebook workflows.
  • Academic use: using Moodle makes the portfolio feel like a real institutional integration rather than a standalone demo.
  • Clear boundary: Examination Middleware remains a middleware system while Moodle remains the official learning platform.

Why Not Alternatives

Custom LMS

A custom LMS would dramatically increase scope: courses, enrollments, faculty dashboards, gradebook rules, and permissions. Examination Middleware only needs to bridge scanned submissions into an LMS, not build the LMS itself.

Google Classroom

Google Classroom is useful for classroom workflows, but it offers less control for custom backend-driven submission mapping, self-hosted academic integration, and LMS-gradebook simulation in this project.

Project Implementation

  • API calls: FastAPI delegates Moodle communication to a Moodle client/service instead of embedding HTTP logic in route handlers.
  • Token auth: Moodle API tokens should be read from environment variables and never exposed to the frontend.
  • Assignment mapping: subject codes or course identifiers from the scanned-paper metadata are mapped to Moodle assignment IDs.
  • Submission state: database records track whether the Moodle call is pending, submitted, failed, or retrying.
  • Response handling: Moodle response IDs or errors are stored for audit logs and staff troubleshooting.

API Request: POST To Moodle

moodle_client.py 
async def save_submission(self, assignment_id: int, item_id: int, token: str | None = None):
    client = await self._get_client()
    params = {
        "wstoken": token or self.token,
        "wsfunction": "mod_assign_save_submission",
        "moodlewsrestformat": "json",
        "assignmentid": str(assignment_id),
        "plugindata[files_filemanager]": str(item_id),
    }
    response = await client.post(f"{self.base_url}/webservice/rest/server.php", data=params)
    response.raise_for_status()
    return response.json()

What it does: sends a POST request to Moodle's REST endpoint using the required web-service parameters, including the assignment ID and the uploaded draft file item ID.

Why it is important: this is the core external API boundary. Examination Middleware does not write directly into Moodle's database; it integrates through Moodle's official service layer.

Token Usage

moodle_client.py 
async def get_site_info(self, token: str | None = None):
    client = await self._get_client()
    ws_token = token or self.token
    params = {
        "wstoken": ws_token,
        "wsfunction": "core_webservice_get_site_info",
        "moodlewsrestformat": "json",
    }
    response = await client.post(f"{self.base_url}/webservice/rest/server.php", data=params)
    response.raise_for_status()
    return response.json()

What it does: uses the Moodle web-service token on each request as `wstoken`, which lets the backend act on behalf of the authenticated LMS user without exposing credentials again.

Why it is important: token-based integration keeps the Moodle boundary stateless and secure. Every request is independently authorized by Moodle using the provided token.

Submission Logic

submission_service.py 
async def _execute_submission(self, artifact, assignment_id: int, moodle_token: str, lock_submission: bool):
    client = MoodleClient(token=moodle_token)
    upload_result = await client.upload_file(file_path=artifact.file_blob_path, token=moodle_token, filename=artifact.original_filename)
    item_id = upload_result["itemid"]
    await client.get_submission_status(assignment_id=assignment_id, token=moodle_token)
    save_result = await client.save_submission(assignment_id=assignment_id, item_id=item_id, token=moodle_token)
    status_result = await client.get_submission_status(assignment_id=assignment_id, token=moodle_token)
    if lock_submission and status_result["lastattempt"].get("cansubmit", False):
        await client.submit_for_grading(assignment_id=assignment_id, token=moodle_token)
    return {"item_id": item_id, "save_result": save_result}

What it does: orchestrates the full Moodle handoff: upload to draft area, verify assignment access, save the submission, re-check status, and optionally finalize for grading.

Why it is important: Moodle submission is not one API call. This service layer is what turns a verified scanned paper into a reliable LMS submission workflow.

Retry Logic

submission_service.py 
except MoodleAPIError as e:
    should_queue = self._should_queue_for_retry(e)
    await self.artifact_service.mark_failed(
        artifact_id=artifact.id,
        error_message=str(e),
        queue_for_retry=should_queue,
    )
    if should_queue:
        return False, "Submission queued - Moodle is temporarily unavailable", {
            "queued": True, "error": str(e)
        }
    return False, f"Submission failed: {e.message}", {"error": str(e)}

What it does: catches Moodle API failures, classifies transient errors such as maintenance or connection issues, marks the artifact accordingly, and queues safe retries instead of losing the submission attempt.

Why it is important: external APIs fail. This retry-aware path is what makes the Moodle integration production-oriented rather than a fragile one-shot HTTP call.

Data Flow

[ Verified Submission ] -> [ Assignment Mapping ] -> [ Moodle Token Auth ] -> [ API Upload ] -> [ LMS Response ] -> [ Sync Log ]
  • The middleware remains responsible for mapping and retry state; Moodle is responsible for the final LMS assignment and gradebook layer.
  • This separation makes failures easier to diagnose because the sync log can show whether mapping, upload, or Moodle response handling failed.

Performance

  • API latency handling: Moodle calls should use timeouts so a slow LMS does not block FastAPI workers indefinitely.
  • Retry logic: transient failures should move the submission to a retrying state rather than forcing staff or students to repeat the entire workflow.
  • Queue handoff: high-volume exam sessions should enqueue Moodle submissions and process them in the background.
  • Batch strategy: future versions can batch multiple submissions by subject or assignment when LMS rate limits require it.

Security

  • Moodle API tokens must be stored server-side in environment variables.
  • Students must not be allowed to choose arbitrary Moodle assignment IDs from the frontend.
  • Assignment mapping should be validated against database subject/course records.
  • Logs should store response status and correlation IDs, not raw LMS tokens or sensitive payloads.
  • Submission ownership must be checked before any Moodle upload is attempted.

Limitations

  • External dependency: Moodle availability, API configuration, token validity, and assignment setup directly affect submission success.
  • Configuration complexity: real Moodle instances require correct roles, web services, permissions, course IDs, and assignment mappings.
  • Latency variance: Moodle response time may be slower than internal database operations, so the integration must be timeout and retry aware.
  • API coupling: Moodle API changes or permission changes can break the integration if not isolated in a client layer.

Final Justification

Moodle is the right integration target because Examination Middleware is an examination middleware system, not a replacement LMS. Moodle provides the academic submission and grading context, while FastAPI and PostgreSQL handle validation, mapping, retry state, and auditability before data reaches the LMS.
