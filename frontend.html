UI + Integration Layer

Frontend (HTML/CSS/JS)

Lightweight interface layer for staff uploads, student dashboard interactions, live demo navigation, and backend API communication.

Role In ExamFlow Nexus

  • The frontend is the UI layer used by staff and students to interact with the examination middleware.
  • Staff use it to open the upload portal and submit scanned answer-sheet files.
  • Students use it to log in, verify identity, view mapped submissions, and trigger LMS submission workflows.
  • The portfolio pages also use the same frontend stack to explain architecture, credentials, live demos, and tech documentation.

Architecture

[ UI ] -> [ API Calls ] -> [ Backend ] -> [ Response ] -> [ Render ]
UI-> API Calls-> Backend-> Response-> Render

UI Workflow

[ User Action ] -> [ Fetch ] -> [ Display ] -> [ Submit ] -> [ Update UI ]
  • User Action: staff selects a file or student opens dashboard/submission controls.
  • Fetch: JavaScript sends a request to a FastAPI endpoint.
  • Display: the UI renders upload status, dashboard records, errors, or copy feedback.
  • Submit: user confirms an action such as upload or Moodle submission.
  • Update UI: DOM state changes to show success, failure, pending, or retry information.

Why This Stack

  • Lightweight: HTML/CSS/JS avoids a large frontend bundle for a project whose UI is focused and workflow-driven.
  • Simple deployment: static assets can be served with the FastAPI app or opened directly for the portfolio site.
  • Fast iteration: form events, fetch requests, copy buttons, and responsive diagrams are straightforward without a build pipeline.
  • Clear integration: the frontend remains a thin client while backend validation, authentication, and data integrity stay server-side.

Why Not React / Angular

React

React would be useful for a large real-time dashboard, but for this project it adds build tooling, component state, and bundle complexity without enough UI complexity to justify it.

Angular

Angular is excellent for enterprise-scale frontends, but it is too heavy for a staff upload page, student dashboard, copy actions, and portfolio documentation pages.

Project Implementation

  • Fetch API: sends login, dashboard, upload, and submit requests to FastAPI endpoints.
  • DOM rendering: updates status labels, submission lists, credential copy feedback, and error messages without a framework.
  • Responsive CSS: grid and flex layouts keep architecture diagrams, cards, and forms readable on mobile.
  • Progressive behavior: copy buttons, menu toggles, and scroll animations enhance the page without blocking core content.

Code Snippet: Fetch Example

async function loadStudentDashboard(token) {
  const response = await fetch("/student/dashboard", {
    headers: { Authorization: `Bearer ${token}` },
  });
  if (!response.ok) throw new Error("Dashboard load failed");
  return response.json();
}

uploadForm.addEventListener("submit", async (event) => {
  event.preventDefault();
  const formData = new FormData(uploadForm);
  const result = await fetch("/upload", { method: "POST", body: formData });
  statusLabel.textContent = result.ok ? "Upload queued" : "Upload failed";
});

Integration Data Flow

[ Staff / Student UI ] -> [ fetch() ] -> [ FastAPI Route ] -> [ JSON Response ] -> [ DOM Update ]
  • The frontend does not decide whether a student can access a paper; it only displays the backend response after JWT and database checks pass.
  • This keeps UI code simple and prevents security rules from living only in the browser.

Performance

  • Minimal load: no framework runtime means fewer bytes and faster first render.
  • Fast rendering: direct DOM updates are enough for small lists, status badges, and form feedback.
  • Static caching: CSS, JS, images, and documentation pages can be cached by the browser.
  • Low JavaScript cost: interactions are event-driven rather than constantly polling or re-rendering large component trees.

Security Considerations

  • Never store production secrets or Moodle tokens in frontend JavaScript.
  • Use backend JWT validation as the real access boundary.
  • Sanitize or safely render backend-provided values to avoid injecting unsafe markup into the DOM.
  • Display user-friendly errors without exposing stack traces, database details, or internal file paths.

Limitations

  • Scalability for large UI: if the student dashboard grows into a complex real-time application, vanilla DOM management can become harder to maintain.
  • Repeated markup: multiple static documentation pages duplicate layout unless a templating system is added.
  • Manual state management: larger form workflows may eventually benefit from a component framework or state library.

Final Justification

HTML, CSS, and vanilla JavaScript are the right frontend choice for ExamFlow Nexus because the UI is focused on clear staff/student workflows, simple API integration, responsive documentation, and fast deployment. The backend carries the complex validation and security, while the frontend stays lightweight and easy to demonstrate.
