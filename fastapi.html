Core Backend Engine

FastAPI

Async API engine powering exam-paper upload, validation, student dashboard access, queue-ready processing, and Moodle LMS submission.

Role In System

  • FastAPI acts as the core backend engine for ExamFlow Nexus.
  • It sits between the staff/student portals and the backend services that process files, update the database, and submit data to Moodle LMS.
  • It handles API routing for upload, dashboard, and submission endpoints.
  • It validates request data before it reaches file processing or LMS integration logic.
  • It coordinates JWT-protected access so staff-only and student-only workflows remain separated.

Tech Workflow

[ Staff Upload ] -> [ Request ] -> [ Validation ] -> [ Auth ] -> [ File Processing ] -> [ DB ] -> [ Queue ] -> [ Moodle ] -> [ Response ]
  • Staff Upload: staff sends a scanned answer sheet through the upload portal.
  • Request: FastAPI receives the multipart request at the upload route.
  • Validation: file metadata, expected fields, and route parameters are checked.
  • Auth: JWT dependency confirms the user is allowed to upload or submit.
  • File Processing: service layer parses filename metadata and stores the artifact.
  • DB: submission state is inserted or updated for dashboard visibility.
  • Queue: retry-ready submission state is prepared for Moodle handoff.
  • Moodle: LMS client submits verified data.
  • Response: FastAPI returns a clean status payload to the portal.

Why FastAPI

  • Async performance: the system performs I/O-heavy work such as uploads, database access, and Moodle HTTP calls. FastAPI supports async handlers so slow external operations do not block unrelated requests.
  • Pydantic validation: request models and typed dependencies reduce invalid student IDs, malformed payloads, and unsafe submission states before service logic runs.
  • API-first design: ExamFlow Nexus is a middleware platform, not just a rendered website. FastAPI fits route-driven integration work better than a minimal microframework.
  • Dependency injection: authentication, database sessions, and service dependencies can be composed cleanly per endpoint.

Why Not Alternatives

Flask

Flask is lightweight, but this project would need more manual validation, more custom auth wiring, and less natural async support for Moodle API calls and upload workflows.

Django

Django is powerful for full-stack applications, but it is too heavy for a focused middleware service whose main responsibilities are API routing, validation, service orchestration, and LMS integration.

Project Implementation

  • /upload: accepts scanned answer sheets from staff, validates access, and sends the file into the artifact processing pipeline.
  • /student/dashboard: loads mapped submissions for a verified student and prevents cross-student access.
  • /submit: moves a verified submission toward Moodle LMS through the submission service and queue/retry path.
  • Service layer: keeps route handlers thin by moving file parsing, database writes, and Moodle calls into dedicated backend services.

Code Snippet: FastAPI Route

@router.post("/upload")
async def upload_exam_file(
    file: UploadFile,
    staff=Depends(require_staff),
    db: Session = Depends(get_db),
):
    artifact = await file_processor.store_and_parse(file)
    submission = submission_service.create(db, artifact, staff.id)
    return {"submission_id": submission.id, "status": submission.status}

Code Snippet: Dependency Injection

def require_staff(payload: dict = Depends(validate_jwt)):
    if payload.get("role") != "staff":
        raise HTTPException(status_code=403, detail="Staff access required")
    return payload

Code Snippet: Async Submission

@router.post("/submit")
async def submit_to_lms(
    submission_id: int,
    student=Depends(require_student),
    db: Session = Depends(get_db),
):
    result = await submission_service.submit_to_moodle(db, submission_id, student["sub"])
    return {"status": result.status, "moodle_ref": result.reference}

Performance

  • Async I/O: upload handling and Moodle submission can be implemented with async functions so network-bound work does not freeze the API worker.
  • Concurrency: multiple students can load dashboards while staff upload or Moodle submission requests are waiting on I/O.
  • Queue-ready design: the route can respond after recording a submission state while the retry manager handles LMS delivery.
  • Thin route handlers: pushing processing into services keeps request handling predictable and easier to profile.

Security

  • Middleware: common request handling can enforce CORS, headers, logging, and error boundaries.
  • JWT validation: protected routes decode and verify tokens before staff upload or student dashboard logic runs.
  • Role authorization: staff routes reject student tokens, and student routes verify ownership before returning submission records.
  • Input validation: request fields and uploaded files are checked before database writes or Moodle calls.
  • Safe error handling: responses should not expose file paths, JWT secrets, Moodle tokens, or internal stack traces.

Limitations

  • Learning curve: developers must understand async functions, dependency injection, and typed request models.
  • Async complexity: blocking file or database operations inside async routes can reduce the benefits of concurrency.
  • Background work: FastAPI alone is not a full job queue; heavy file processing and LMS retries should move to background workers or a queue manager as traffic grows.
  • Operational discipline: route boundaries, service modules, and error contracts must stay clean as the middleware expands.

Final Justification

FastAPI is the strongest backend fit for ExamFlow Nexus because the system is API-first, validation-heavy, I/O-bound, and integration-focused. It gives the middleware a clean way to receive staff uploads, protect student workflows with JWT, coordinate file and database services, and submit verified records to Moodle LMS without the overhead of a full-stack framework.
