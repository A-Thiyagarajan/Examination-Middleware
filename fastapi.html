Core Backend Engine

FastAPI

Async API engine powering exam-paper upload, validation, student dashboard access, queue-ready processing, and Moodle LMS submission.

Role In System

  • FastAPI acts as the core backend engine for ExamFlow Nexus.
  • It sits between the staff/student portals and the backend services that process files, update the database, and submit data to Moodle LMS.
  • It handles API routing for upload, dashboard, and submission endpoints.
  • It validates request data before it reaches file processing or LMS integration logic.
  • It coordinates JWT-protected access so staff-only and student-only workflows remain separated.

Tech Workflow

[ Staff Upload ] -> [ Request ] -> [ Validation ] -> [ Auth ] -> [ File Processing ] -> [ DB ] -> [ Queue ] -> [ Moodle ] -> [ Response ]
  • Staff Upload: staff sends a scanned answer sheet through the upload portal.
  • Request: FastAPI receives the multipart request at the upload route.
  • Validation: file metadata, expected fields, and route parameters are checked.
  • Auth: JWT dependency confirms the user is allowed to upload or submit.
  • File Processing: service layer parses filename metadata and stores the artifact.
  • DB: submission state is inserted or updated for dashboard visibility.
  • Queue: retry-ready submission state is prepared for Moodle handoff.
  • Moodle: LMS client submits verified data.
  • Response: FastAPI returns a clean status payload to the portal.

Why FastAPI

  • Async performance: the system performs I/O-heavy work such as uploads, database access, and Moodle HTTP calls. FastAPI supports async handlers so slow external operations do not block unrelated requests.
  • Pydantic validation: request models and typed dependencies reduce invalid student IDs, malformed payloads, and unsafe submission states before service logic runs.
  • API-first design: ExamFlow Nexus is a middleware platform, not just a rendered website. FastAPI fits route-driven integration work better than a minimal microframework.
  • Dependency injection: authentication, database sessions, and service dependencies can be composed cleanly per endpoint.

Why Not Alternatives

Flask

Flask is lightweight, but this project would need more manual validation, more custom auth wiring, and less natural async support for Moodle API calls and upload workflows.

Django

Django is powerful for full-stack applications, but it is too heavy for a focused middleware service whose main responsibilities are API routing, validation, service orchestration, and LMS integration.

Project Implementation

  • /upload: accepts scanned answer sheets from staff, validates access, and sends the file into the artifact processing pipeline.
  • /student/dashboard: loads mapped submissions for a verified student and prevents cross-student access.
  • /submit: moves a verified submission toward Moodle LMS through the submission service and queue/retry path.
  • Service layer: keeps route handlers thin by moving file parsing, database writes, and Moodle calls into dedicated backend services.

Technical Code Showcase

This path is the heart of the backend: FastAPI receives the request, validates the upload, hands work to services, and returns a structured response the portal can render immediately.

API Route: Upload Endpoint

upload.py 
@router.post("/single", response_model=FileUploadResponse)
async def upload_single_file(file: UploadFile = File(...), exam_session: str = Form(...),
    db: AsyncSession = Depends(get_db), current_staff: StaffUser = Depends(get_current_staff)):
    content = await file.read()
    is_valid, message, metadata = file_processor.validate_file(content, file.filename)
    if not is_valid:
        return FileUploadResponse(success=False, message=message, errors=[message])
    file_path, file_hash = await file_processor.save_file(content, file.filename, "pending")
    artifact = await artifact_service.create_artifact(file_blob_path=file_path, file_hash=file_hash, parsed_reg_no=metadata["parsed_register_no"])
    return FileUploadResponse(success=True, artifact_uuid=str(artifact.artifact_uuid), workflow_status=artifact.workflow_status.value)

What it does: accepts a multipart file upload from staff, reads the scanned paper asynchronously, validates the filename and file type, then persists the file before returning structured metadata.

Why it is important: this is the first entry point in the pipeline, so it proves FastAPI can handle file input, dependency injection, validation, and JSON response generation in one clean request cycle.

Dependency Injection: JWT-Protected Staff Access

auth.py 
async def get_current_staff(
    token: str = Depends(oauth2_scheme),
    db: AsyncSession = Depends(get_db),
) -> StaffUser:
    payload = decode_access_token(token)
    if payload is None or payload.get("type") != "staff":
        raise HTTPException(status_code=status.HTTP_401_UNAUTHORIZED, detail="Could not validate credentials")
    result = await db.execute(select(StaffUser).where(StaffUser.id == int(payload["sub"])))
    staff = result.scalar_one_or_none()
    if staff is None or not staff.is_active:
        raise HTTPException(status_code=status.HTTP_403_FORBIDDEN, detail="Account is disabled")
    return staff

What it does: resolves the current staff user through FastAPI dependencies, decodes the bearer token, checks role/type claims, and loads the active staff record from the database.

Why it is important: it keeps authorization outside the route body, which makes protected endpoints smaller, easier to test, and much harder to bypass accidentally.

Service Layer: File Parsing And Validation

file_processor.py 
def validate_file(self, file_content: bytes, filename: str):
    metadata = {"original_filename": filename, "size_bytes": len(file_content)}
    if len(file_content) > settings.max_file_size_bytes:
        return False, "File too large", metadata
    ext = os.path.splitext(filename)[1].lower()
    if ext not in settings.allowed_extensions_list:
        return False, "Invalid file type", metadata
    register_no, subject_code, is_parsed = self.parse_filename(filename)
    metadata.update({"parsed_register_no": register_no, "parsed_subject_code": subject_code})
    if not is_parsed:
        return False, "Invalid filename format. Expected: REGISTER_SUBJECT.pdf", metadata
    return True, "File validated successfully", metadata

What it does: validates upload size, allowed extensions, and the register-number-plus-subject filename convention, then returns normalized metadata for downstream storage and mapping.

Why it is important: this is where the backend converts raw scanned files into trusted domain data, which prevents malformed papers from reaching the database or Moodle submission flow.

Async Processing: Moodle Submission Workflow

submission_service.py 
async def submit_artifact(self, artifact_uuid: str, moodle_token: str, register_number: str):
    artifact = await self.artifact_service.get_by_uuid(artifact_uuid)
    assignment_id = await self._resolve_assignment_id(artifact)
    try:
        result = await self._execute_submission(artifact=artifact, assignment_id=assignment_id, moodle_token=moodle_token, lock_submission=True)
        await self.artifact_service.mark_submitted(artifact_id=artifact.id, moodle_submission_id=result.get("submission_id"))
        return True, "Submission completed successfully", result
    except MoodleAPIError as e:
        should_queue = self._should_queue_for_retry(e)
        await self.artifact_service.mark_failed(artifact_id=artifact.id, error_message=str(e), queue_for_retry=should_queue)
        return False, "Submission queued - Moodle is temporarily unavailable", {"queued": should_queue}

What it does: awaits assignment resolution, runs the Moodle submission workflow asynchronously, marks success on completion, and falls back to retry-aware failure handling for transient LMS errors.

Why it is important: it demonstrates the production path after upload: the backend can continue from validated artifact to Moodle delivery without blocking the API design on synchronous external calls.

Performance

  • Async I/O: upload handling and Moodle submission can be implemented with async functions so network-bound work does not freeze the API worker.
  • Concurrency: multiple students can load dashboards while staff upload or Moodle submission requests are waiting on I/O.
  • Queue-ready design: the route can respond after recording a submission state while the retry manager handles LMS delivery.
  • Thin route handlers: pushing processing into services keeps request handling predictable and easier to profile.

Security

  • Middleware: common request handling can enforce CORS, headers, logging, and error boundaries.
  • JWT validation: protected routes decode and verify tokens before staff upload or student dashboard logic runs.
  • Role authorization: staff routes reject student tokens, and student routes verify ownership before returning submission records.
  • Input validation: request fields and uploaded files are checked before database writes or Moodle calls.
  • Safe error handling: responses should not expose file paths, JWT secrets, Moodle tokens, or internal stack traces.

Limitations

  • Learning curve: developers must understand async functions, dependency injection, and typed request models.
  • Async complexity: blocking file or database operations inside async routes can reduce the benefits of concurrency.
  • Background work: FastAPI alone is not a full job queue; heavy file processing and LMS retries should move to background workers or a queue manager as traffic grows.
  • Operational discipline: route boundaries, service modules, and error contracts must stay clean as the middleware expands.

Final Justification

FastAPI is the strongest backend fit for ExamFlow Nexus because the system is API-first, validation-heavy, I/O-bound, and integration-focused. It gives the middleware a clean way to receive staff uploads, protect student workflows with JWT, coordinate file and database services, and submit verified records to Moodle LMS without the overhead of a full-stack framework.
